# Documentation Change Workflow

Use for README, architecture, runbook, policy, map, workflow, or skill documentation updates.

## Process

1. Read the relevant existing docs.
2. Preserve existing conventions and tone.
3. Prefer updating existing docs over creating new ones unless a new workflow/runbook/map file is the clearer structure.
4. Do not invent commands, contracts, or repo facts.
5. If docs and code conflict, trust code and report the mismatch.
6. Validate links, paths, command names, and config references where practical.

## Handoff

Report:

- files changed
- facts inferred vs assumed
- any missing user input
- validation performed or skipped
